# Product Requirements Document (PRD)

## 1. Обзор продукта
- **Название:** Family Hub
- **Цель:** Единое семейное приложение для совместного управления задачами, покупками, бюджетом, рецептами, проектами и идеями — с реалтайм-синхронизацией между членами семьи.
- **Целевая аудитория:** Семейная пара (двое взрослых), совместное ведение быта. В будущем — дети, расширенная семья, потенциальная монетизация.

## 2. Роли пользователей

| Роль | Описание | Права |
|------|----------|-------|
| Член семьи | Любой пользователь, привязанный к family_id | Полный CRUD во всех модулях, назначение задач, просмотр данных всей семьи |

> В MVP одна роль — все равны. Роли (admin, ребёнок, гость) — v2.

#### AUTH

##### US-001: Регистрация и вход
- **Как** член семьи, **я хочу** войти по email/паролю, **чтобы** получить доступ к данным моей семьи
- **Acceptance Criteria:**
  - GIVEN незарегистрированный пользователь WHEN вводит email + пароль на странице регистрации THEN создаётся аккаунт и профиль с новым family_id
  - GIVEN зарегистрированный пользователь WHEN вводит email + пароль THEN попадает на главную страницу
  - GIVEN неверный пароль WHEN попытка входа THEN показывается ошибка, аккаунт не блокируется
  - GIVEN авторизованный пользователь WHEN сессия истекла THEN перенаправляется на страницу входа

##### US-002: Приглашение в семью
- **Как** член семьи, **я хочу** пригласить партнёра по ссылке/коду, **чтобы** мы были привязаны к одному family_id
- **Acceptance Criteria:**
  - GIVEN авторизованный пользователь WHEN генерирует invite-код THEN код действителен 24 часа
  - GIVEN новый пользователь WHEN регистрируется по invite-коду THEN привязывается к существующему family_id
  - GIVEN invite-код WHEN истёк срок THEN код не работает, показывается сообщение

##### US-003: Выход из аккаунта
- **Как** член семьи, **я хочу** выйти из аккаунта, **чтобы** защитить данные на общем устройстве
- **Acceptance Criteria:**
  - GIVEN авторизованный пользователь WHEN нажимает "Выйти" THEN сессия уничтожается, перенаправление на логин

##### US-004: Профиль пользователя
- **Как** член семьи, **я хочу** указать своё имя и аватар, **чтобы** партнёр видел кто создал/выполняет задачу
- **Acceptance Criteria:**
  - GIVEN авторизованный пользователь WHEN открывает профиль THEN видит имя, email, аватар
  - GIVEN пользователь WHEN меняет display_name THEN имя обновляется везде в интерфейсе в реальном времени

#### МОДУЛЬ "ЗАДАЧИ"

##### US-010: Создание задачи
- **Как** член семьи, **я хочу** быстро создать задачу, **чтобы** не забыть что нужно сделать
- **Acceptance Criteria:**
  - GIVEN авторизованный пользователь WHEN заполняет форму (название обязательно, описание, дедлайн, приоритет, список, назначение — опционально) THEN задача создаётся и видна всем членам семьи
  - GIVEN пустое название WHEN попытка сохранить THEN ошибка валидации, задача не создаётся
  - GIVEN новая задача WHEN создана THEN статус = "Новая", автор = текущий пользователь

##### US-011: Редактирование и удаление задачи
- **Как** член семьи, **я хочу** изменить или удалить задачу, **чтобы** держать список актуальным
- **Acceptance Criteria:**
  - GIVEN существующая задача WHEN пользователь меняет любое поле THEN изменения сохраняются и видны в realtime
  - GIVEN существующая задача WHEN пользователь удаляет её THEN задача исчезает у всех членов семьи

##### US-012: Статусы задач
- **Как** член семьи, **я хочу** менять статус задачи, **чтобы** видеть прогресс
- **Acceptance Criteria:**
  - GIVEN задача со статусом "Новая" WHEN пользователь отмечает чекбокс THEN статус = "Выполнена"
  - GIVEN задача со статусом "Выполнена" WHEN пользователь снимает чекбокс THEN статус = "Новая"
  - GIVEN задача WHEN пользователь меняет статус через меню THEN доступны: Новая, В работе, Выполнена, Архив
  - GIVEN Kanban-доска WHEN пользователь перетаскивает карточку в другую колонку THEN статус меняется автоматически

##### US-013: Kanban-доска
- **Как** член семьи, **я хочу** видеть задачи в виде Kanban-доски, **чтобы** видеть картину целиком
- **Acceptance Criteria:**
  - GIVEN вид "Kanban" WHEN открыт THEN 4 колонки: Новые, В работе, Выполненные, Архив
  - GIVEN карточка задачи WHEN перетаскивается (drag & drop) в другую колонку THEN статус меняется
  - GIVEN мобильное устройство WHEN пользователь тянет карточку пальцем THEN drag & drop работает корректно
  - GIVEN карточка WHEN отображается THEN видно: название, приоритет (цвет), дедлайн, на ком задача

##### US-014: Список задач
- **Как** член семьи, **я хочу** видеть задачи списком, **чтобы** быстро пробежаться по ним
- **Acceptance Criteria:**
  - GIVEN вид "Список" WHEN открыт THEN задачи группируются по статусу (секции сворачиваемые)
  - GIVEN секция WHEN пользователь сворачивает/разворачивает THEN состояние сохраняется

##### US-015: Календарь задач
- **Как** член семьи, **я хочу** видеть задачи в календаре, **чтобы** планировать по дням
- **Acceptance Criteria:**
  - GIVEN вид "Календарь" WHEN открыт THEN задачи с дедлайном отображаются на соответствующих днях
  - GIVEN день в календаре WHEN пользователь нажимает на него THEN видит список задач на этот день
  - GIVEN задача без дедлайна WHEN вид "Календарь" THEN задача не отображается (только в Kanban/Списке)

##### US-016: Приоритеты
- **Как** член семьи, **я хочу** ставить приоритет задаче, **чтобы** фокусироваться на важном
- **Acceptance Criteria:**
  - GIVEN форма задачи WHEN выбирается приоритет THEN доступны: Высокий (красный), Средний (жёлтый), Низкий (серый)
  - GIVEN список/kanban WHEN задача с высоким приоритетом THEN визуально выделена цветом

##### US-017: Дедлайны
- **Как** член семьи, **я хочу** ставить дату и время дедлайна, **чтобы** не пропустить срок
- **Acceptance Criteria:**
  - GIVEN форма задачи WHEN пользователь выбирает дату THEN открывается календарь-пикер
  - GIVEN форма задачи WHEN пользователь выбирает время THEN открывается time-пикер (часы + минуты)
  - GIVEN задача с дедлайном WHEN текущее время > дедлайн THEN задача помечается как просроченная (красный индикатор)

##### US-018: Списки задач (группировка)
- **Как** член семьи, **я хочу** группировать задачи по спискам, **чтобы** разделить "Дом", "Работа", "Дети"
- **Acceptance Criteria:**
  - GIVEN пользователь WHEN создаёт новый список THEN список появляется в фильтре
  - GIVEN задача WHEN создаётся/редактируется THEN можно выбрать список из dropdown
  - GIVEN фильтр по списку WHEN выбран список THEN показываются только задачи этого списка

##### US-019: Делегирование задач
- **Как** член семьи, **я хочу** назначить задачу на партнёра, **чтобы** было понятно кто за что отвечает
- **Acceptance Criteria:**
  - GIVEN форма задачи WHEN назначение THEN выбор из списка членов семьи (аватар + имя)
  - GIVEN задача назначена WHEN отображается в списке/kanban THEN видно аватар исполнителя
  - GIVEN задача WHEN не назначена THEN считается общей (видна всем, нет конкретного исполнителя)
  - GIVEN фильтр "Мои задачи" WHEN активен THEN показываются только задачи, назначенные на текущего пользователя

##### US-020: Подзадачи (чеклист)
- **Как** член семьи, **я хочу** добавить чеклист внутри задачи, **чтобы** разбить сложную задачу на шаги
- **Acceptance Criteria:**
  - GIVEN открытая задача WHEN пользователь добавляет подзадачу THEN она появляется как чекбокс внутри задачи
  - GIVEN подзадачи WHEN часть выполнена THEN показывается прогресс (3/5)
  - GIVEN все подзадачи выполнены WHEN пользователь смотрит на задачу THEN визуальный индикатор "все шаги готовы"

##### US-021: Повторяющиеся задачи
- **Как** член семьи, **я хочу** создать повторяющуюся задачу, **чтобы** не создавать её вручную каждый раз
- **Acceptance Criteria:**
  - GIVEN форма задачи WHEN включает повторение THEN доступны: ежедневно, еженедельно, ежемесячно, кастом (выбор дней)
  - GIVEN повторяющаяся задача WHEN выполнена THEN автоматически создаётся следующая на следующий период
  - GIVEN повторяющаяся задача WHEN удалена THEN спрашивает: "Удалить только эту" или "Удалить все будущие"

##### US-022: Комментарии к задаче
- **Как** член семьи, **я хочу** оставить комментарий к задаче, **чтобы** обсудить детали с партнёром
- **Acceptance Criteria:**
  - GIVEN открытая задача WHEN пользователь пишет комментарий THEN он появляется с именем автора и временем
  - GIVEN новый комментарий от партнёра WHEN задача открыта THEN комментарий появляется в realtime

##### US-023: Теги/метки
- **Как** член семьи, **я хочу** добавлять теги к задачам, **чтобы** фильтровать по темам
- **Acceptance Criteria:**
  - GIVEN форма задачи WHEN пользователь вводит тег THEN тег сохраняется (создаётся если новый, или выбирается из существующих)
  - GIVEN список задач WHEN фильтр по тегу THEN показываются только задачи с этим тегом
  - GIVEN тег WHEN отображается THEN цветной бейдж рядом с названием задачи

##### US-024: Спринты
- **Как** член семьи, **я хочу** группировать задачи в спринты, **чтобы** планировать работу на период
- **Acceptance Criteria:**
  - GIVEN настройки WHEN пользователь создаёт спринт THEN указывает название, дату начала, длительность (по умолчанию 2 недели)
  - GIVEN спринт WHEN активен THEN задачи можно привязать к нему
  - GIVEN спринт WHEN просмотр THEN показывается прогресс: сколько задач выполнено из общего числа
  - GIVEN спринт WHEN завершён THEN незавершённые задачи можно перенести в следующий спринт или бэклог

##### US-025: Напоминания
- **Как** член семьи, **я хочу** получить напоминание о задаче, **чтобы** не пропустить дедлайн
- **Acceptance Criteria:**
  - GIVEN задача с дедлайном WHEN пользователь включает напоминание THEN доступны: за 1 час, за 1 день, в момент дедлайна
  - GIVEN время напоминания наступило WHEN приложение открыто THEN показывается уведомление в интерфейсе
- **Примечания:** Push/Telegram уведомления — v2. В MVP напоминания только в интерфейсе (in-app).

##### US-026: Realtime-синхронизация задач
- **Как** член семьи, **я хочу** видеть изменения задач партнёра мгновенно, **чтобы** мы работали с актуальными данными
- **Acceptance Criteria:**
  - GIVEN два устройства WHEN один пользователь создаёт/изменяет/удаляет задачу THEN второй видит изменение без перезагрузки
  - GIVEN два пользователя WHEN одновременно меняют разные поля одной задачи THEN оба изменения сохраняются (нет конфликта)

#### МОДУЛЬ "ПОКУПКИ"

##### US-030: Создание списка покупок
- **Как** член семьи, **я хочу** создать список покупок (напр. "Сільпо"), **чтобы** группировать товары по магазинам
- **Acceptance Criteria:**
  - GIVEN пользователь WHEN создаёт новый список THEN указывает название, список появляется как вкладка
  - GIVEN пустой список WHEN открыт THEN показывается пустое состояние с подсказкой

##### US-031: Добавление товара
- **Как** член семьи, **я хочу** быстро добавить товар в список, **чтобы** не забыть что купить
- **Acceptance Criteria:**
  - GIVEN открытый список WHEN пользователь вводит название товара THEN товар добавляется мгновенно (optimistic UI)
  - GIVEN товар WHEN добавляется THEN можно указать количество (опционально)
  - GIVEN скорость добавления WHEN пользователь на ходу THEN добавление занимает < 3 секунды

##### US-032: Отметка "Куплено"
- **Как** член семьи, **я хочу** отметить товар как купленный, **чтобы** видеть что осталось
- **Acceptance Criteria:**
  - GIVEN товар в списке WHEN пользователь нажимает чекбокс THEN товар зачёркивается и уходит в низ списка
  - GIVEN купленный товар WHEN пользователь снимает чекбокс THEN товар возвращается в активные

##### US-033: Удаление товара с Undo
- **Как** член семьи, **я хочу** удалить товар с возможностью отмены, **чтобы** не терять данные по ошибке
- **Acceptance Criteria:**
  - GIVEN товар WHEN удалён THEN показывается toast "Удалено" с кнопкой "Отменить" на 4 секунды
  - GIVEN toast WHEN пользователь нажимает "Отменить" THEN товар восстанавливается
  - GIVEN toast WHEN 4 секунды прошли THEN товар удаляется окончательно

##### US-034: Управление списками
- **Как** член семьи, **я хочу** переименовать или удалить список, **чтобы** держать порядок
- **Acceptance Criteria:**
  - GIVEN список WHEN переименовывается THEN новое название отображается во вкладке
  - GIVEN список WHEN удаляется THEN подтверждение: "Удалить список X и N товаров?"

##### US-035: Офлайн-режим покупок
- **Как** член семьи, **я хочу** чекать товары без интернета в магазине, **чтобы** список работал в подвале
- **Acceptance Criteria:**
  - GIVEN нет интернета WHEN пользователь открывает список покупок THEN список отображается из кеша
  - GIVEN нет интернета WHEN пользователь чекает товар THEN чекбокс срабатывает локально
  - GIVEN интернет восстановился WHEN есть локальные изменения THEN они синхронизируются с сервером

##### US-036: Realtime-синхронизация покупок
- **Как** член семьи, **я хочу** видеть что партнёр добавил в список, **чтобы** не покупать дважды
- **Acceptance Criteria:**
  - GIVEN два устройства WHEN один добавляет/чекает/удаляет товар THEN второй видит мгновенно
  - GIVEN optimistic UI + realtime WHEN оба срабатывают THEN нет дублей (дедупликация)

#### МОДУЛЬ "ИДЕИ"

##### US-040: Создание идеи
- **Как** член семьи, **я хочу** быстро записать идею, **чтобы** не потерять мысль
- **Acceptance Criteria:**
  - GIVEN пользователь WHEN создаёт идею THEN заполняет: заголовок (обязательно), описание (опционально)
  - GIVEN идея WHEN создана THEN отображается в общем списке идей с датой и автором

##### US-041: Теги и категории идей
- **Как** член семьи, **я хочу** помечать идеи тегами, **чтобы** потом быстро найти нужную
- **Acceptance Criteria:**
  - GIVEN идея WHEN добавляется/редактируется THEN можно добавить теги
  - GIVEN список идей WHEN фильтр по тегу THEN показываются только идеи с этим тегом

##### US-042: Конвертация идеи
- **Как** член семьи, **я хочу** превратить идею в задачу или проект, **чтобы** начать действовать
- **Acceptance Criteria:**
  - GIVEN идея WHEN пользователь нажимает "→ Задача" THEN создаётся задача с заголовком и описанием идеи
  - GIVEN идея WHEN пользователь нажимает "→ Проект" THEN создаётся проект с названием и описанием идеи
  - GIVEN идея конвертирована WHEN просмотр THEN идея помечается как "Конвертирована в задачу/проект" (не удаляется)

##### US-043: Поиск идей
- **Как** член семьи, **я хочу** искать по идеям, **чтобы** найти записанное ранее
- **Acceptance Criteria:**
  - GIVEN список идей WHEN пользователь вводит текст в поиск THEN фильтрация по заголовку и описанию

#### МОДУЛЬ "РЕЦЕПТЫ"

##### US-050: Создание рецепта
- **Как** член семьи, **я хочу** сохранить рецепт, **чтобы** не искать его каждый раз
- **Acceptance Criteria:**
  - GIVEN пользователь WHEN создаёт рецепт THEN заполняет: название, описание, ингредиенты (список), инструкции (текст)
  - GIVEN рецепт WHEN сохранён THEN отображается в списке рецептов

##### US-051: Фото рецепта
- **Как** член семьи, **я хочу** прикрепить фото к рецепту, **чтобы** видеть как блюдо выглядит
- **Acceptance Criteria:**
  - GIVEN форма рецепта WHEN пользователь загружает фото THEN фото сохраняется и отображается в карточке
  - GIVEN рецепт без фото WHEN отображается THEN показывается placeholder-иконка

##### US-052: Ингредиенты → Покупки
- **Как** член семьи, **я хочу** отправить ингредиенты рецепта в список покупок, **чтобы** не переписывать вручную
- **Acceptance Criteria:**
  - GIVEN рецепт с ингредиентами WHEN пользователь нажимает "В покупки" THEN выбирается целевой список покупок
  - GIVEN целевой список выбран WHEN подтверждение THEN ингредиенты добавляются как товары в список

##### US-053: Поиск рецептов
- **Как** член семьи, **я хочу** найти рецепт по названию, **чтобы** быстро открыть нужный
- **Acceptance Criteria:**
  - GIVEN список рецептов WHEN поиск THEN фильтрация по названию и ингредиентам

#### МОДУЛЬ "ПРОЕКТЫ"

##### US-060: Создание проекта
- **Как** член семьи, **я хочу** создать проект для крупной цели, **чтобы** структурировать шаги
- **Acceptance Criteria:**
  - GIVEN пользователь WHEN создаёт проект THEN заполняет: название, описание, заметки
  - GIVEN проект WHEN создан THEN статус = "В работе"

##### US-061: Подзадачи проекта
- **Как** член семьи, **я хочу** добавить подзадачи в проект, **чтобы** видеть прогресс
- **Acceptance Criteria:**
  - GIVEN проект WHEN пользователь добавляет подзадачу THEN указывает: название, статус, стоимость (опционально), ссылка (опционально)
  - GIVEN подзадачи WHEN часть выполнена THEN прогресс-бар обновляется (3/10 = 30%)

##### US-062: Бюджет проекта
- **Как** член семьи, **я хочу** указать стоимость элементов проекта, **чтобы** видеть общий бюджет
- **Acceptance Criteria:**

## 4. User Flows

### Flow 1: Регистрация и подключение партнёра
1. Пользователь А открывает приложение → видит страницу регистрации
2. Вводит email + пароль → создаётся аккаунт + профиль + новый family_id
3. Попадает на главную (пусто, подсказки "Создай первую задачу")
4. Открывает настройки → "Пригласить в семью" → генерируется invite-код
5. Отправляет код партнёру (любым способом: мессенджер, лично)
6. Пользователь Б регистрируется по invite-коду → привязывается к тому же family_id
7. Оба видят общие данные

### Flow 2: Планирование дня (Задачи)
1. Пользователь открывает "Задачи" → видит Kanban-доску
2. Нажимает FAB (+) → форма создания задачи
3. Вводит название, выбирает список "Дом", приоритет "Высокий", дедлайн — сегодня 18:00, назначает на партнёра
4. Сохраняет → задача появляется в колонке "Новые"
5. Партнёр видит задачу мгновенно (realtime)
6. Партнёр перетаскивает в "В работе" → начинает выполнять
7. По завершении — чекает → задача в "Выполнена"

### Flow 3: Поход в магазин (Покупки)
1. Дома: Пользователь открывает "Покупки" → вкладка "Сільпо"
2. Добавляет: Хліб, Молоко, Яйця (быстро, по одному, без лишних полей)
3. Партнёр видит список в realtime, добавляет: Сир, Масло
4. В магазине: Пользователь открывает список (даже без интернета в подвале)
5. Чекает товары по мере покупки → зачёркиваются
6. При выходе из магазина (есть интернет) → изменения синхронизируются

### Flow 4: Рецепт → Покупки
1. Пользователь открывает "Рецепты" → находит "Борщ"
2. Нажимает "В покупки" → выбирает список "Сільпо"
3. Ингредиенты (буряк, капуста, картопля...) добавляются в список покупок
4. Идёт в магазин — всё в одном списке

### Flow 5: Управление бюджетом
1. Пользователь открывает "Бюджет" → видит баланс за месяц
2. Добавляет доход: "Зарплата", 35000 UAH
3. Настраивает регулярные платежи: "Коммуналка" 3000 UAH/мес, "Netflix" 250 UAH/мес
4. Когда приходит время платить → видит напоминание
5. Отмечает "Оплачено" → записывается в расходы, дата сдвигается на следующий месяц
6. В конце месяца: видит сколько ушло, сколько осталось

### Flow 6: Идея → Проект
1. Приходит мысль: "Сделать детскую площадку во дворе"
2. Открывает "Идеи" → быстро записывает заголовок + описание
3. Через неделю: решают действовать → "→ Проект"
4. Создаётся проект с названием и описанием
5. Добавляет подзадачи: "Купить песочницу — 2500 UAH", "Купить горку — 5000 UAH"
6. Видит общий бюджет проекта: 7500 UAH
7. По мере выполнения — прогресс-бар растёт

## 10. Открытые вопросы

- Точная цветовая палитра — решит Design Agent

## Handoff → Architect Agent
- **Критичные требования:** Модульная архитектура (6 модулей + расширяемость). Свой PostgreSQL + auth (JWT/bcrypt) + WebSocket realtime. PWA с офлайном для покупок. Делегирование задач между пользователями одной семьи.
- **Нагрузка:** 2-5 пользователей, до 10 000 записей на семью, realtime < 500ms, API < 200ms.
- **Интеграции:** Нет внешних в MVP. В v2: Telegram Bot, n8n, OCR.
- **Риски:** WebSocket на одном сервере (без managed service). Офлайн-синк (конфликты). Спринты + повторяющиеся задачи — сложная логика в БД.
